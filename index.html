Kubernetes Deep Dive
Architecture, Pods, Flow — from your handwritten notes
🏗️
What is Kubernetes?
Orchestration ToolManages containers across multiple machines
Auto ScalingAutomatically scales containers on heavy load or crashes
Self HealingAutomatically restarts crashed containers
Load DistributionSplits containers across machines equally
Server ScalingServers increase automatically as demand grows
🖥️
Master Node Components
API ServerEvery communication is done through API Server — central hub
Controller Manager (CM)Maintains desired state of containers. Checks desired vs actual state
ETCDDatabase — stores desired state and actual state of all resources
SchedulerDecides which Worker Node a newly created container should run on
⚙️
Worker Node Components
KubeletWatches container state. If container crashes, informs API Server
Container Runtime (CRM)Actually creates and runs containers (e.g. containerd, Docker)
ContainersApplication containers running inside Pods on Worker Nodes
Kube-proxyHandles networking rules and Pod-to-Pod communication
🔁
Pod Concepts
PodSmallest unit in K8s. Wraps one or more containers with shared storage & network
Pod IPEach Pod gets a unique IP. Pods communicate when they know each other's Pod IP
Storage in PodUsed to persist data beyond container lifecycle
Multi-container PodOnly for tightly coupled applications that must share resources
Pod CrashIf container in pod crashes, entire pod is deleted and recreated
Kubernetes Self-Healing Flow
How K8s recovers from a container crash — step by step
Recovery Flow
1
Kubelet watches container state — detects crash
2
Kubelet informs API Server about the crash
3
API Server notifies Controller Manager to check state
4
API Server queries ETCD for desired state vs actual state
5
Controller Manager compares desired vs actual — finds mismatch
6
API Server asks Scheduler — which Worker Node should host the new container?
7
API Server instructs Kubelet on the chosen node to create container
8
CRM (Container Runtime) creates the container for the image
Create Pod Flow
1
Write a pod.yml manifest file with apiVersion, kind, metadata, spec
2
Run kubectl apply -f pod.yml to send manifest to API Server
3
API Server stores desired state in ETCD
4
Scheduler decides which node gets the Pod
5
Kubelet on that node creates the container via CRM
6
Use kubectl get pods or kubectl describe pods pod-name to verify
Pod YAML Structure
📄
pod.yml Template
apiVersion: v1 # API version kind: Pod # Type of resource metadata: name: pod-name # Name of the pod labels: app: my-app # Info about pods spec: containers: - name: container-name image: nginx:latest ports: - containerPort: 80
kubectl apply -f pod.ymlCreate pod from YAML file
kubectl get pods -o wideList pods with node and IP info
kubectl describe pods pod-nameDetailed pod info and events
kubectl delete pod pod-nameDelete a specific pod

Git & GitHub — Interview Q&A
What is Git?
Git is a distributed version control system used to track changes in source code and coordinate work among multiple developers.
What is GitHub?
GitHub is a cloud-based platform that hosts Git repositories and provides collaboration features like pull requests, issue tracking, and CI/CD integration.
What are Conventional Commit types?
fix: msg — Fix a bug/error
feat: msg — Adding a new feature
refactor: msg — Changing variable names/structure
docs: msg — Documentation changes
chore: msg — Changes that don't affect logic
!fix: msg — Breaking change / makes error
What is a Pull Request?
A Pull Request is used to push modifications when you don't have direct access to the main branch. In the repo's PR tab, maintainers can see and review the request before merging.
What is git stash?
git stash saves your current uncommitted changes temporarily. git stash pop restores those files back to the working directory.
What is a Merge Conflict?
A merge conflict occurs when two branches edit the same lines. Steps to resolve: (1) git pull — shows the conflict markers, (2) resolve manually and merge the code.
What is GitHub Actions?
GitHub Actions is a CI/CD solution focused on GitHub. It is not multi-platform dependent and automates GitHub workflows using code snippets (YAML workflows).
Git Workflow Summary

→ git commit -m "message"
→ git remote add origin (url)
→ git branch -M main
→ git push origin main
git log — shows all commits in repo. git diff branchname — checks difference between main and that branch.
Docker — From My Notes
Why do we need Docker?
Without Docker, a web application (PHP + MySQL + Apache) runs on one machine. Moving it to another machine means re-installing all software. Docker packages the app + software into a container — platform independent.
Web App + PHP + MySQL + Code + Software → Container → Platform Independent
What is a Docker Image?
A Docker Image is a template/snapshot of an application. Flow: Software Application → Dockerfile → Image. The image can be stored locally or on DockerHub.

Where does Docker run?
Docker runs on Linux natively. On Windows and MacOS it uses Hyper-V / WSL2 (Hyperkit on Mac) to allow other OS to run within their OS environment.

AWS Key Concepts
What is EC2?
EC2 (Elastic Compute Cloud) is an AWS service to create virtual machines in the cloud. VMs act like real servers — install OS, runtime, app code. Commonly used to host dynamic websites running backend apps like Node.js, Python, Java.
EC2 provides virtual machines in the cloud and is commonly used to host dynamic websites by running backend applications and server-side logic.
What is EBS and EBS Snapshot?
EBS volumes are virtual cloud disks that provide persistent storage for EC2 instances and can be backed up using snapshots.

An EBS Snapshot is a point-in-time backup of an EBS volume — incremental, durable, and can be copied across regions for disaster recovery.
Public vs Private Subnets
Public subnet → inbound + outbound (IGW) — for apps reachable by users
Private subnet → outbound only (NAT) — for deployment and backend processing
Database → no internet at all (most secure)
Public subnets provide internet access mainly for application availability to users, while deployment and backend processing are usually handled in private subnets.
Final Project: EBS Snapshot Automation
Designed an automated solution to manage and delete stale EBS snapshots in AWS for cloud cost optimization. Used AWS Lambda + Python + Boto3 to connect to EC2 APIs, identify orphaned snapshots, and delete them automatically. Used IAM roles for secure access without exposing credentials.
Lambda function deletes snapshots not linked to any volume, whose volumes are detached, or whose volumes are deleted. Secure via IAM roles. Efficient via serverless execution.
